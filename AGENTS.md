# NUEDC 嵌入式工程开发规范

适用工程：`2026_Diansai` —— MSPM0G3519（LQFP-100，SDK 2.11.00.07，ticlang 4.0.4），由旧工程 `NUEDC`（MSPM0G3507）移植而来，移植配方见 `agent/MIGRATION_G3507_TO_G3519.md`。唯一硬件配置源为仓库根 `board.syscfg`；固件构建入口为 `rtk make -C Debug all`。与 G3507 时期的差异：编码器为 TIMG8/TIMG9 硬件 QEI（GROUP1 中断仅服务按键）、步进总线物理实例为 UART7、MPU6050/I2C_IMU 已移除、灰度传感器为 12 路。

本文件是本仓库所有新增、修改、重构和评审工作的强制规范。任何实现都必须先满足这里定义的架构边界，再考虑功能完成度。若用户的临时要求与本规范冲突，必须停止编码，说明冲突和可行的分层方案，不得先写出破坏架构的代码再补救。

## 1. 核心原则

1. **单向依赖**：只允许上层依赖下层，下层严禁包含、调用或感知上层。
2. **模块化**：一个模块只负责一种清晰能力，接口表达能力，不暴露实现机制。
3. **解耦**：模块通过头文件声明的最小接口和显式数据结构通信，不通过跨模块全局变量、隐藏副作用或上层回调通信。
4. **功能归位**：代码必须放在最窄且真正拥有该职责的层中，不得以“改动方便”为理由跨层放置。
5. **测试先行**：先定义可验证行为并编写测试，再完成最小实现；修复缺陷时必须先复现缺陷。
6. **小步修改**：只修改当前目标需要的内容，不夹带无关重构，不复制已有的违规依赖。

## 2. 编码前必须回答的问题

开始设计或编码前，必须简要回答以下三个问题；答案不清楚时不得进入实现：

1. **抽象是什么？** 外部真正需要的能力是什么，而不是内部寄存器、协议帧或状态机细节是什么。
2. **什么必须隐藏？** 私有状态、缓冲区、索引、硬件句柄、寄存器、协议解析状态和控制参数中，哪些不得由调用者直接操作。
3. **代码属于哪一层？** Driver、Middleware、App Service 或 App Task 中，谁拥有该职责。

随后必须先定义最小稳定接口，写清参数约束、返回含义、调用前置条件、资源所有权和失败方式，再编写实现。

## 3. 目标架构

本项目的目标架构从底层到上层如下：

```text
App Task / Scheduler / UI
           |
           v
       App Service
        /        \
       v          v
   Middleware   Driver
                   |
                   v
                DL HAL
```

箭头表示源码依赖和主动调用方向。数据可以由 Driver 产生，但上层必须主动查询、读取或消费；Driver 不得通过调用上层函数来反向传递控制权。

### 3.1 DL HAL 层

DL HAL 包括：

- TI DriverLib，即 `ti/driverlib/*`。
- SysConfig 生成接口和符号，包括 `ti_msp_dl_config.h`。
- 芯片寄存器、NVIC、DMA、GPIO、Timer、UART、I2C 等硬件细节。
- 与具体 MSPM0 型号、板级引脚和外设实例绑定的代码。

DL HAL 是最低层，不得包含任何 `hc-team` 上层模块。除 Driver 层实现文件外，其他层不得直接包含 DL HAL 头文件或操作寄存器。

### 3.2 Driver 层

目录：`hc-team/driver/`

Driver 是唯一允许调用 DL HAL 的项目代码层，负责：

- 外设和器件的初始化、读写、启停与状态查询。
- 板级资源映射、协议收发、超时和硬件错误转换。
- ISR、DMA、环形缓冲区、事件标志和硬件资源所有权。
- 隐藏寄存器、SysConfig 宏、硬件句柄、协议帧及内部状态。

Driver 的公共头文件必须优先只暴露标准 C 类型、Driver 自有枚举和不透明接口，不得暴露 DL HAL 类型、寄存器定义或私有结构体布局。

Driver 不得：

- 包含或调用 Middleware、App Service、App Task、Scheduler 或 UI。
- 引用任何上层符号，即使通过 `extern`、弱符号或函数指针间接引用也不允许。
- 接收上层注册的回调并在 Driver 或 ISR 中调用该回调。
- 实现业务策略、控制目标、页面逻辑或任务状态机。

### 3.3 Middleware 层

目录：`hc-team/middleware/`

Middleware 是纯平台无关的数据处理和算法层，负责 PID、滤波、坐标变换、数据解析、估计器等可脱离硬件运行的逻辑。

Middleware 必须：

- 使用自有且与硬件无关的输入、输出、配置和状态类型。
- 通过参数、返回值或输出结构接收和输出数据。
- 能在主机测试环境中编译和运行，不要求真实 MCU 外设。
- 将算法状态封装在模块私有状态或由调用者显式持有的上下文中。

Middleware 不得：

- 包含或调用 Driver、DL HAL、App 的任何代码。
- 出现引脚、外设实例、寄存器、IRQ、DMA 通道或 SysConfig 宏。
- 主动读取传感器、发送电机命令、刷新 UI 或参与任务调度。
- 通过跨层全局变量获取输入或发布结果。

### 3.4 App Service 层

目录：`hc-team/app/service/`

Service 是具体功能层，也是 Driver 数据与 Middleware 算法之间唯一的业务编排桥。每个 Service 应围绕一项完整能力组织，例如循迹、速度闭环、二维平台或设备诊断。

Service 负责：

- 主动从 Driver 查询或读取数据。
- 将 Driver 数据转换为 Middleware 自有输入结构。
- 调用 Middleware 完成算法处理。
- 校验算法输出并调用 Driver 应用结果。
- 管理功能级初始化、启动、停止、周期更新、状态和错误恢复。
- 隐藏功能内部使用了哪些 Driver 和 Middleware。

Service 公共接口应优先采用以下能力形式：

```c
Status Service_Init(void);
Status Service_Start(const ServiceConfig *config);
Status Service_Update(void);
Status Service_Stop(void);
Status Service_GetState(ServiceState *state);
```

示例仅表示接口职责，不要求所有模块使用相同命名或无条件增加五个函数。必须按实际需求保留最小接口，不得预先构造无需求的通用框架。

### 3.5 App Task 层

目录：`hc-team/app/tasks/`

Task 是调度层，只负责“何时运行”和“运行哪个功能”，包括周期触发、任务启停、运行状态推进以及调用 Service。

Task 必须保持短小，不得承载算法、协议组包、硬件读写或复杂业务实现。Task 原则上只能调用 Service 和 App 调度接口，不得直接调用 Driver、Middleware 或 DL HAL。

### 3.6 App 其他子层

- `app/scheduler/`：只负责系统状态、运行项、任务组和生命周期调度，不实现设备操作或业务算法。
- `app/system/`：作为系统装配入口，负责按依赖顺序启动模块并进入调度；不得直接包含 DL HAL。板级初始化必须由 Driver 接口提供。
- `app/ui/`：只负责输入意图、页面模型和显示编排；不得直接操作显示硬件或业务设备，应调用对应 Service 或 App 接口。

### 3.7 Utils

`hc-team/utils/` 只允许存放真正跨领域、无硬件和无业务含义的纯工具。Utils 不得依赖 Driver、Middleware 或 App，也不得成为绕开分层的公共杂物目录。仅被单一模块使用的工具必须留在该模块内部。

## 4. 依赖矩阵

`允许`表示该行可以主动包含或调用该列；`禁止`表示不得产生源码依赖。

| 调用方 | DL HAL | Driver | Middleware | Service | Task/Scheduler/UI |
|---|---:|---:|---:|---:|---:|
| DL HAL | 内部 | 禁止 | 禁止 | 禁止 | 禁止 |
| Driver | 允许 | 同层受控 | 禁止 | 禁止 | 禁止 |
| Middleware | 禁止 | 禁止 | 同层受控 | 禁止 | 禁止 |
| App Service | 禁止 | 允许 | 允许 | 同层受控 | 禁止 |
| App Task/Scheduler/UI | 禁止 | 禁止 | 禁止 | 允许 | 同层受控 |

同层依赖也必须保持明确方向，禁止循环包含和循环调用。两个模块需要彼此了解时，必须重新划分职责或提取一个更低层、稳定且确有必要的接口。

以下判定没有例外：

- Middleware 包含 Driver 或 DL HAL：**失败**。
- Driver 包含或调用 App：**失败**。
- App Task 直接包含 Driver、Middleware 或 DL HAL：**失败**。
- Service 拉取 Driver 数据，转换后按值传给 Middleware：**通过**。
- Driver 在 ISR 中缓存数据，Service 随后主动读取：**通过**。

## 5. Driver 与 Middleware 的桥接规则

标准数据流必须是：

```text
Driver 采集/缓存硬件数据
        ^
        | Service 主动读取
        |
Service 转换为算法输入 -> Middleware 处理 -> Service 校验输出
                                                |
                                                v
                                      Driver 执行硬件命令
```

禁止 Driver 到上层的注册回调。需要异步处理时必须采用以下方式之一：

1. ISR 更新 Driver 私有事件标志，上层通过 `HasEvent`/`GetStatus` 一类接口查询。
2. ISR 将数据写入 Driver 私有环形缓冲区，上层通过 `Read`/`TryRead`/`Consume` 接口读取。
3. Driver 维护只增计数器或状态快照，上层通过原子快照接口获取。

ISR 只能完成必要的中断状态读取与清除、最小数据搬运、计数或置位。解析、算法、日志、UI、阻塞等待和业务状态转换必须移到 Service 或 Task 上下文。

## 6. 接口、状态与资源所有权

- 公共 API 必须说明用途、参数范围、返回值、调用上下文、前置条件和失败结果。
- 私有状态必须使用模块内 `static` 或不透明上下文隐藏，不得在公共头文件暴露可被任意修改的内部结构。
- 禁止一个模块直接读写另一个模块的全局变量；共享可写全局状态视为架构错误。
- 跨模块传递指针或缓冲区时，必须明确所有者、有效期、容量、可变性以及调用后是否仍可访问。
- 默认使用不可变输入：能使用 `const` 的输入必须使用 `const`；更新状态时不得偷偷修改调用者未授权的数据。
- 共享于 ISR 与主循环的数据必须明确 `volatile`、原子性和临界区策略。`volatile` 不能替代原子操作或并发保护。
- 有限资源必须有唯一所有者；UART、DMA 通道、Timer、I2C 总线和设备实例不得被多个模块无协调地配置。
- 状态机必须记录状态、事件、转换条件、动作和非法转换处理；复杂状态机应提供转换表或紧邻实现的说明。

## 7. 错误处理与安全

- 所有外部输入、通信数据、长度、索引、枚举和配置必须在系统边界校验。
- 所有硬件等待必须有明确超时，不得无界轮询。
- 错误不得静默吞掉；必须返回明确状态，或记录足够诊断上下文并进入已定义的安全状态。
- Driver 必须把底层错误转换为本模块稳定的错误类型，不得迫使上层理解寄存器位。
- 禁止硬编码密码、令牌、私钥或其他秘密；发现秘密泄露必须停止工作并报告轮换需求。
- 禁止越界访问、未经校验的长度运算、不受控格式化、悬空指针和未初始化状态。
- 未经测量不得声称或实施性能优化。优化前必须记录基线、目标和验证方法。

## 8. AI 编码常见风险与强制检查

AI 编写代码时不得只阅读当前文件或当前模块。开始修改前必须检查该功能的上游数据来源、下游调用者、初始化路径、停止路径和硬件执行结果。搜索范围至少覆盖相关 Driver、Middleware、Service、Task、调度注册和所有调用点。

### 8.1 电机与功率器件保护

涉及直流电机、步进电机、舵机、TB6612、H 桥、MOSFET、继电器或其他可能损坏硬件的执行器时，硬件保护是功能的一部分，不得省略。任何电机代码在首次输出前必须确认 PWM 频率、有效极性、方向映射、计数周期、占空比单位和硬件允许范围，禁止根据函数名或旧代码猜测这些参数。

电机控制必须满足：

- 上电、复位、初始化未完成、任务退出和错误状态下，默认输出必须为停止或硬件定义的安全态。
- 速度、占空比、频率、加速度和位置命令必须在唯一负责层做明确限幅；不得在多个层重复限幅并产生不一致结果。
- 输出变化必须有符合机构能力的斜率或加速度限制，禁止单周期从大正值跳到大负值。
- 正反转切换必须先降到零，确认进入停止区间，等待硬件所需死区后再反向。不得通过快速交替方向制造脉冲震荡。
- H 桥输入组合、刹车、滑行、待机和使能顺序必须依据实际芯片与接线定义；禁止生成可能造成桥臂直通的组合。
- 周期控制必须检测命令或反馈是否过期；超时后必须停止输出，不得保持最后一次大功率命令。
- 必须提供简单、确定且可从正常控制流调用的停止接口。保护动作不得依赖调试器暂停程序。
- 修改 PID、编码器方向、PWM、定时器时钟、任务周期或电机接线定义时，必须重新审查完整电机链路，不能只验证被修改的模块。

首次硬件验证必须分阶段进行：先断开负载或架空机构，以低占空比、低速度和短持续时间确认方向、PWM 频率及停止行为，再逐步增加输出。必须使用串口、OLED、示波器、逻辑分析仪或直接可观察的硬件行为留下验证结果；不得把连接调试器或单步观察作为验收方式。

若无法确认电机参数、方向、保护时序或安全输出，必须停止并询问，不得生成可直接驱动硬件的猜测代码。

### 8.2 防止重复数据处理

修改采样、传感器、编码器、Middleware、PID 或控制输出前，必须阅读上下游 API 的实际实现和全部调用点，不能只看头文件。必须先记录该数据的处理链：

```text
原始来源 -> Driver 采样/校准 -> Service 单位与语义转换
         -> Middleware 算法 -> Service 输出转换 -> Driver 执行
```

对链路中的每个值必须确认：

- 单位和数值尺度，例如脉冲、转数、RPM、弧度、角度、毫米、秒或毫秒。
- 正负方向、坐标系、零点和机械安装方向。
- 数据是瞬时值、累计值、增量值还是已经计算的速度。
- 是否已经反向、归一化、限幅、去抖、校准、滤波、积分或微分。
- 采样周期、更新时间、有效期以及由谁拥有和更新。

同一种数据变换必须只有一个明确所有者。除非算法设计明确要求多级滤波并有参数与测试依据，否则禁止在 Driver、Service 和 Middleware 中重复执行方向反转、滤波、缩放、积分、微分、限幅或单位换算。例如，编码器 Driver 已完成方向统一或滤波时，PID 输入路径不得再次进行同类处理。

修改数据生产者时必须搜索并审查所有消费者；修改消费者时也必须追溯到原始生产者。若无法证明输入数据当前处于什么处理阶段，不得继续添加新的数据处理。

### 8.3 精简实现，拒绝无依据的防御代码

本项目优先追求清晰、可验证的嵌入式功能，不按企业级通用框架堆叠抽象、状态码和防御分支。实现必须是解决当前需求的最小方案，不得为了显得“健壮”而增加大量无实际用途的 `if`、错误码、包装层、配置项或恢复状态机。

每个保护检查必须同时满足：

1. 对应一个真实、可描述的故障模型。
2. 失败后有明确且有用的处理动作。
3. 能通过测试或硬件现象验证该分支。

不满足以上条件的检查应删除或不引入。调用者不会处理的细分错误码不得层层传播；优先使用满足当前控制流需要的最小状态集合。对于模块内部由静态结构保证、调用路径完全受控且不可能为空的对象，不得机械地在每一层重复空指针检查。

精简不等于删除必要保护。以下检查始终保留：外部输入边界、数组和缓冲区长度、硬件超时、资源冲突、电机安全态、可能损坏硬件的输出限制，以及已有真实故障证据覆盖的检查。

本项目开发与验收默认不使用调试器，任何设计都不得依赖断点、单步或内存窗口才能确认正确性。不得引入只有调试器才能理解和确认的复杂安全状态；验证必须依靠可重复测试、构建结果、串口/OLED 输出、测量仪器和明确的硬件现象。

## 9. 强制开发流程

所有功能、缺陷修复和重构必须按顺序执行：

1. **全局链路检视**：搜索相关模块、上下游实现、全部调用点、初始化和停止路径；涉及数据时记录单位与已完成处理，涉及电机时记录完整控制链路。
2. **架构归属检查**：回答三个设计问题，确认依赖矩阵允许该设计。
3. **定义验收标准**：把需求转换为可观察、可重复的通过条件，并列出边界和失败路径。
4. **最小接口设计**：先定义稳定头文件契约和数据所有权，不暴露实现细节。
5. **测试先行（RED）**：先写会因缺少功能或现有缺陷而失败的测试或可重复验证步骤。
6. **最小实现（GREEN）**：只写使验证通过所需的代码。
7. **改进（IMPROVE）**：消除当前改动引入的重复、无依据检查和复杂度，保持所有测试通过。
8. **分层与链路评审**：检查调用方向、数据是否被重复处理、全局状态、ISR、超时、输入校验、资源所有权和电机安全。
9. **完整验证**：运行单元测试、集成测试、构建、静态依赖扫描和必要的硬件冒烟测试。
10. **交付说明**：报告修改、验证结果、未执行的硬件测试、假设和剩余风险。

不得通过修改正确的测试来掩盖实现缺陷。若测试本身错误，必须先说明错误依据，再修正测试。

## 10. 测试与验收要求

测试是交付内容，不是可选收尾工作。

- **单元测试**：覆盖 Middleware 算法、Service 状态转换、数据转换、边界值和错误路径。
- **集成测试**：覆盖 Service 与 Driver/Middleware 的契约、缓冲区消费、超时和恢复流程。
- **端到端/硬件测试**：覆盖关键任务进入、运行、停止，真实外设收发以及故障后的安全行为。
- 可在主机运行的 Middleware 和 Service 代码，覆盖率目标不得低于 80%。无法纳入主机覆盖率的 HAL、ISR 和硬件路径必须用集成测试、断言、逻辑分析或硬件冒烟记录提供证据。
- 每个修复必须保留一个能够防止同类回归的测试或确定性复现步骤。
- 涉及采样或算法链路时，必须验证单位、方向、采样周期以及每项滤波/换算只在预定层执行一次。
- 涉及电机时，必须验证上电默认停止、限幅、换向过零与死区、命令超时停止、任务退出停止和错误状态停止。

提交前至少执行并记录：

1. 当前 CCS/Make 构建或项目规定的等价构建。
2. 相关单元与集成测试及覆盖率。
3. 对非法跨层 `#include` 和符号依赖的搜索。
4. 编译警告、静态分析和输入安全检查。
5. `git diff` 审查，确认没有秘密、生成物或无关修改。
6. 涉及硬件行为时的硬件冒烟清单；无法执行时必须明确说明。
7. 对新增保护分支逐项确认其故障模型、处理动作和验证证据，删除无依据的防御代码。

## 11. 存量架构债务

当前仓库仍存在与目标架构不一致的存量代码，包括但不限于：

- Driver 反向依赖 App 调度符号。
- App 文件直接包含 `ti_msp_dl_config.h` 或 `ti/driverlib/*`。
- Task 直接操作 Driver 或 Middleware，并承载部分业务逻辑。
- `app/service/` 尚未完全承接现有业务编排。
- Driver 使用上层注册回调传递 UART、DMA 或其他事件。
- 主机测试套件 `tests/host/` 未随 G3507→G3519 迁移带入本仓库（旧套件在 `../NUEDC/tests/host/`）。历史验收记录引用的主机测试无法在本仓库复跑；下一次施工开工前必须先迁入该套件并恢复全绿基线（拓扑登记 V16）。

这些问题是待迁移债务，不代表它们是新代码可采用的范例。本规范落地时不要求一次性重构全部存量代码，但必须遵守：

1. 新代码不得增加或复制上述违规模式。
2. 修改涉及违规代码时，必须在当前需求允许的最小范围内向目标架构迁移。
3. 迁移必须小步进行，保持现有行为，并为每一步提供构建、测试或硬件验证。
4. 不得借修复一个局部问题之名重写无关模块。
5. 暂时无法消除的违规依赖必须在交付说明中列出原因、影响和后续迁移边界。

## 12. 停止并报告的条件

出现以下任一情况时，必须停止实现并先报告架构冲突：

- 业务逻辑需要直接操作寄存器或 DL HAL。
- Middleware 需要直接读取 Driver 数据或控制硬件。
- Driver 需要调用 App、Service、Task 或上层注册回调。
- 一个模块需要直接修改另一个模块的内部状态或全局变量。
- 公共头文件需要暴露寄存器、私有缓冲区或内部状态机布局。
- 需求只能通过循环依赖、无界等待或绕过输入校验完成。
- 性能改动没有可用基线和测量方法。
- 电机代码缺少可确认的频率、方向、限幅、换向或安全停止条件。
- 数据进入当前模块前是否已经反向、滤波、换算或限幅无法确认。

报告时必须给出冲突点、违反的层级规则，以及至少一个保持单向依赖的替代方案。

## 13. 完成定义

只有同时满足以下条件，任务才算完成：

- 功能和失败路径符合验收标准。
- 依赖方向符合本文件矩阵，没有新增跨层违规。
- 接口最小、状态隐藏、资源所有权明确。
- 测试先行证据、相关测试和构建均通过。
- 关键硬件路径已验证，或明确记录未验证原因和风险。
- 没有新增编译警告、安全问题、硬编码秘密或无关改动。
- 已审查完整上下游链路，没有重复的数据处理；涉及电机时已完成安全项检查。
- 交付说明包含假设、验证命令、结果和剩余风险。

提交信息必须使用 Conventional Commits：`<type>: <description>`，其中 `type` 使用 `feat`、`fix`、`refactor`、`docs`、`test`、`chore`、`perf` 或 `ci`。

## 14. API 架构拓扑强制维护

本项目的当前 API 平台架构拓扑位于：

```text
agent/api_architecture_topology.md
```

任何代码新增、修改、删除或重构都必须执行以下流程：

1. **编码前先读拓扑**：找到待修改模块的 API 类、所有上游调用者、下游依赖和数据流；对照本文件依赖矩阵检查是否存在交叉依赖。
2. **编码前查完整链路**：从数据来源追到最终硬件或输出，确认单位、方向、滤波、校准和限幅由谁负责，不能只看当前文件。
3. **编码后更新拓扑**：同步修改 Mermaid 类图、逻辑图、API、依赖箭头、数据处理位置、资源所有权和交叉风险登记。
4. **更新复核日期和日志**：每次代码工作结束都必须在拓扑日志新增记录；即使 API 和依赖没有变化，也必须记录“已复核，无拓扑变化”。
5. **拓扑与代码冲突时以代码事实为准**：先把拓扑修正为当前真实状态，再继续设计；不得为了让图看起来合规而隐藏现有违规依赖。

若一次改动新增 Driver→App、Middleware→Driver、Task→Driver/Middleware、循环包含、跨模块全局写入或重复数据处理，而计划中没有明确迁移步骤，必须停止并报告，不得继续实现。

拓扑更新是完成定义的一部分。代码已修改但拓扑未复核和更新时，任务不得标记为完成。
