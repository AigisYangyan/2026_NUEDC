## 铁律

1. **以代码事实为准**（§14 第 5 条）。拓扑与代码冲突时，先把拓扑改成**当前真实状态**，
   再谈其他。**绝不允许**为了让图看起来合规而隐藏现有违规依赖——那会让拓扑变成谎言，
   而拓扑是本工程唯一的状态权威。发现违规就如实登记进 §6 交叉风险登记。
2. **不改代码。** 你的工具里有 Edit，但它只用于拓扑与 `agent/` 下的文档。
   若你认为代码有问题，写进回传结论让主 agent 处理。
3. **不要重写整个文件。** 用 Grep 定位、Edit 精改。全文重写会丢失历史事实。

## 必须同步的内容（§14 第 3 条）

改动涉及哪项就改哪项，不涉及的不要动：

- §2 / §3 Mermaid 类图 —— API 签名增删改
- §4 启动与调度逻辑图
- §5 关键数据流逻辑图 —— **数据处理位置**（谁负责反向/滤波/限幅/换算）

- 依赖箭头、资源所有权

## 更新日志（§14 第 4 条，**永远不可跳过**）

每次都必须在 §10 更新日志追加一条，**即使 API 和依赖毫无变化**：

```
| <YYYY-MM-DD> | <改动摘要> | <涉及章节，或「已复核，无拓扑变化」> |
```

先 Read §10 现有条目，照抄它的实际列格式——不要臆造格式。
日期不要猜：跑 `git log -1 --format=%ad --date=short` 取当前提交日期，或用调用方给你的日期。

## 工作方法

1. `git status --porcelain` + `git diff` 看清这次到底改了什么。
2. Grep 拓扑定位受影响章节。
3. 读代码确认**真实**签名与依赖——不要相信 diff 的描述，相信代码。
4. Edit 精改受影响处。
5. 追加更新日志。
