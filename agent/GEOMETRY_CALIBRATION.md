## 0. 通用单位与符号约定

- **长度一律 mm。**
- **左右一律驾驶员视角**（沿车头前进方向看）——与既有约定一致（灰度的「左」= 车头的左）。
- 灰度位序：`+误差 = 线在车中心右侧`（M02/track_error 口径）；位序左右的唯一修正点是
  `bit0_is_left`（`track_error.h:34`，`line_follow.h:65` 透传），标定阶段确认 bit0(=PIN_IN1) 是否在车左。
- 编码器方向：AB 相接反在 `driver/encoder/encoder.c s_direction_sign[]` 唯一吸收；
  航向符号在 `Odometry_Config_T.heading_sign`（`odometry.h:36`）唯一吸收。**标定这两处方向时，
  别在里程/距离常量里再翻一次符号**——两处反转互相抵消 = 没修（V22）。

### 1.1 灰度探头间距 `pitch_mm` —— 对应你的 #1 精度

| 项 | 内容 |
|---|---|
| 单一所有者 | `TrackError_Config_T.pitch_mm`（`hc-team/middleware/track_error/track_error.h:33`） |
| 透传字段 | `LineFollow_Config_T.pitch_mm`（`hc-team/app/service/line_follow/line_follow.h:64`） |
| 单位/符号 | mm，正数，**相邻探头感光中心的间距** |
| 消费者 | `TrackError_FromDarkBitmap` → `error_mm`；坐标公式 `探头坐标 = (index − 5.5) × pitch_mm`，12 路 index 0..11、几何中心在 5.5（灰度→循迹链） |
| 怎么测 | 游标卡尺量相邻两探头感光中心距。若阵列严格等距，量首末探头中心距 ÷ (N−1) 取平均更准。**记录是否严格等距**（不等距则循迹解算前提被破坏，须回报）。 |
| 用途说明（#1） | 线位置量化步长 ≈ pitch_mm，`|误差| ≤ 5.5 × pitch_mm` 天然成立。pitch 越小分辨率越细。这就是「灰度精度」的物理来源——精度由**探头间距**决定，不是软件能变的。 |
| 现在能填？ | **待实测**。装配层未写、探头未装，现在无值可填。 |

### 1.2 轮距 `track_width_mm` —— 对应你的 #3「差速」那一半

| 项 | 内容 |
|---|---|
| 单一所有者 | `Motion_Config_T.track_width_mm`（`hc-team/app/service/motion/motion.h:58`） |
| 单位/符号 | mm，正数，**左右驱动轮接地中心的横向距** |

| 怎么测 | 量左右驱动轮**胎面中线**的横向间距（不是轮内侧、不是外侧，是接地点/胎宽中线）。 |
| 用途说明（#3） | 只用于圆弧前馈的内外轮速差。差速循迹本身的差速修正限幅归外环 PID，不归这里。 |
| 现在能填？ | **待实测**。 |

### 1.3 脉冲距 `mm_per_pulse` —— 你的 #3 里被误并进「轮距」的那一半

> **纠正**：高精度**直线距离判断**不靠轮距，靠脉冲距。轮距只管圆弧/差速。
> 两个不同常量、两个不同单一所有者，别合并（合并即踩 V22）。

| 项 | 内容 |
|---|---|
| 单一所有者 | `Odometry_Config_T.mm_per_pulse`（`hc-team/middleware/odometry/odometry.h:35`） |
| 透传字段 | `Motion_Config_T.mm_per_pulse`（`motion.h:50`，装配透传，非第二所有者） |
| 单位/符号 | mm/脉冲，正数 |
| 消费者 | odometry 脉冲→位移：`前进距离 = (Δleft + Δright) × 0.5 × mm_per_pulse`（`odometry.h:80`），里程/航向链 |
| 怎么测 | **必须实跑标定，不要纯几何算**：让车直行一段实测地面距离 D（如 1000 mm），读累计脉冲 P，`mm_per_pulse = D / P`；跑 3~5 次取平均。纯几何（π·轮径 / 线数 / 减速比）只作初值——实测覆盖轮径公差与打滑。 |
| 用途说明 | 这才是「高精度距离判断」的常量。你要的定距直行 / 停位判据都读它。 |
| 现在能填？ | **待实测**。 |

## 2. 无归属：属 T01 段表常量，现在只测不建字段

以下量今天全仓 Grep **零命中**、零消费者。按 S07 裁定「段表由装配层/T01 持有」，
它们是 T01 段表常量，等实车 + 题目角点几何定案时定。**现在建字段 = 投机**。

### 2.1 灰度阵列中心 → 驱动轴纵向距离 `L` —— 你的 #2 核心

**这不是「缺一个常量」，是「压角点转弯 = 组合，不是新原语」。**

- **现状**：无字段、无消费者。`Motion_StartTurn(deg)`（`motion.c:151`）是**绕驱动轴原地差速转、
  靠 IMU 航向闭环**，它没有「把旋转中心前移到灰度阵列」的语义，也**不该**为此加参数。
- **对齐问题的正解**（你的物理直觉正确）：灰度阵列在驱动轴前方 `L`。灰度读到角点时驱动轴
  还在角点后方 `L`，此时原地转会把灰度阵列甩出一条半径 `L` 的弧、转完对不上出线。
  正确动作是**组合**：
  ```
  FOLLOW_UNTIL(角点元素)  →  STRAIGHT(L)  →  TURN(90°)
  灰度读到角      里程计直行 L 把驱动轴推到角点     再原地转，转完灰度落在出线
  ```
  三个原语（元素事件 / 定距直行 / 定角转）**全都已存在**（line_follow 元素事件面 + motion
  STRAIGHT/TURN，S02b/S06 DONE）。不用写新 motion 代码。
- **单位/符号**：mm，正数 = 灰度阵列感光中心线**在驱动轴前方**的纵向距离。
- **怎么测**：量灰度阵列感光中心线到左右驱动轮公共轴线的**纵向垂直距离**。
- **精修提醒**：实测 `L` 只是「直行补偿段」的名义初值。实车角点对齐还受地面、转弯半径、
  元素触发点影响，上板后以段表 `STRAIGHT` 段长微调。**题目角点几何（直角/圆角/线宽）会改这个值**——
  所以定案等题目。
- **落点**：T01 route 段表（`Route_Setup` 注入的段参数）。现在无字段，登记 §5 Q11。
