## 2. 阻断项（必须硬件组答复）

### Q1：封装到底是 LQFP-64 还是 LQFP-100？

- 引脚表标题为「G3519-G3507**兼容**引脚表」，Sheet2 第 6 行明写「统一采用 **LQFP-64(PM)** 物理脚号」，全表脚号按 64 脚编。
- 当前工程 `board.syscfg:4`：`--device "MSPM0G351X" --package "LQFP-100(PZ)"`；`docs/MIGRATION_G3507_TO_G3519.md` 记载迁移目标为 **G3519/LQFP-100**。
- **影响**：封装决定可用引脚集合与 syscfg 校验。若实际板子是 LQFP-64，当前 syscfg 的封装设定即错误，需整体重配；若板子是 LQFP-100，则表的脚号列对本工程无效（**引脚名 PAxx/PBxx 仍可用，脚号列须忽略**）。
- **需要答复**：本次打样的主控板实物封装是哪个？表的脚号列是否仅为 G3507 兼容性参考？

### Q2：陀螺仪走哪个串口——表内两个 sheet 直接打架

| 出处 | 裁定 |
|---|---|
| Sheet1 第 11/12 行 | 陀螺仪 = **UART3**，TX=PA26、RX=PA25 |
| Sheet1 第 41/42 行 | 调试串口 = **UART3**，TX=PA26、RX=PA25，备注"与陀螺仪分离，不再并联" |
| Sheet2 第 4 行 | 陀螺仪4P座 = **PA28(TX→IMU RX)/PA31(RX←IMU TX)**，"保留 PA25/PA26 调试串口" |
| Sheet2 第 5 行 | "运行阶段 UART0 由 PA28/PA31 陀螺仪使用，不能同时作为 VOFA 串口" |

三处矛盾：

1. **Sheet1 内部矛盾**：第 11/12 行与第 41/42 行把**同一对引脚 PA26/PA25、同一个 UART3** 同时判给陀螺仪和调试串口，却备注"不再并联"——物理上不可能。
2. **Sheet1 与 Sheet2 矛盾**：Sheet1 说陀螺仪在 UART3/PA26/PA25；Sheet2 说陀螺仪在 UART0/PA28/PA31。
3. **脚号自相矛盾**：同一个 PA26 在第 11 行标脚号 35、在第 41 行标脚号 30；PA25 在第 12 行标 39、第 42 行标 26。**同一引脚不可能有两个脚号**——表存在数据错误（疑与 Q1 封装混用有关）。

- **当前工程事实**：`UART_IMU` = UART3 PA25/PA26（与 Sheet1 一致）；P5 刚把 IMU 发送迁到该端口（`imu_uart`，提交 `b24a456`）。
- **需要答复**：陀螺仪最终是 **UART3/PA26/PA25** 还是 **UART0/PA28/PA31**？以哪个 sheet 为准？

### Q3：VOFA 遥测串口迁到哪里？

- 当前 `UART_HOST_LINK`（VOFA 上位机遥测）= **UART0 PA10/PA11**。
- 表 Sheet1 第 5/6 行 + Sheet2 第 5 行：PA10/PA11 **仅用于启动阶段 BSL 烧录**，运行阶段 UART0 交给陀螺仪，**明确"不能同时作为 VOFA 串口"**。
- 表未给 VOFA 指定新去处。若按 Sheet1 第 41/42 行把"调试串口"理解为 VOFA，则 VOFA = UART3 PA26/PA25——但那正是 Q2 里陀螺仪也要的端口。
- **影响**：VOFA 是当前唯一的上位机遥测/调参通道（`driver/uart_vofa` + `vofa_register`），删除或停用会直接影响调试能力。
- **需要答复**：VOFA 运行期走哪个 UART/哪对引脚？若确认与陀螺仪共用 UART3，是否接受"调试时插陀螺仪则无遥测"的互斥？

### Q4：编码器与灰度的引脚对调，是否确认按表执行？

这是一次**原子对调**，当前占用完全交叉：

```text
当前 syscfg                      引脚表要求
  QEI_LEFT  = TIMG8, PA7/PA6  →  QEI_LEFT  = TIMG9, PB7/PB9    （PB7/PB9 现为灰度 IN4/IN5）
  QEI_RIGHT = TIMG9, PA3/PA2  →  QEI_RIGHT = TIMG8, PB10/PB11  （PB10/PB11 现为灰度 IN10/IN11）
  灰度 IN4  = PB7             →  灰度 IN4  = PA7               （PA7 现为 QEI_LEFT）
  灰度 IN5  = PB9             →  灰度 IN5  = PB20              （PB20 当前空闲 ✓）
  灰度 IN10 = PB10            →  灰度 IN10 = PB14              （PB14 当前空闲 ✓）
  灰度 IN11 = PB11            →  灰度 IN11 = PB0               （PB0  当前空闲 ✓）
  —                           →  PA6/PA3/PA2 释放
```

Codex 已核实：`PB20`/`PB14`/`PB0`/`PA28`/`PA31` 在当前 `board.syscfg` **零命中（确实空闲）**；`PB7/PB9/PB10/PB11` 当前归灰度、`PA7/PA6/PA3/PA2` 当前归 QEI——对调关系成立且无第三方占用。

表给出的理由可信：Sheet1 第 43 行「PA6…不再用于编码器，**避免核心板晶振功能冲突**」——若属实，这是必须迁的硬理由，而非偏好。

- **⚠ 高风险点（Codex 提请注意）**：本次对调**同时改变了定时器与左右轮的归属**：当前 `QEI_LEFT=TIMG8`、`QEI_RIGHT=TIMG9`；表要求 `左轮=TIMG9`、`右轮=TIMG8`。**若固件只改引脚而不改 timer↔轮 映射，左右轮数据会静默对调**——现象是转向逻辑整体镜像，不报错、不崩溃，极易被误判为 PID 参数问题。此项必须在施工时用 R 行独立证明。
- **⚠ 关联既有裁定**：AB 相接反与左右轮装反在数据上同为符号翻转，**全链路唯一修正点是 `encoder.c:41` `s_direction_sign[] = {-1, 1}`**（该行注释即如此声明）。本次迁移后必须重新标定该常量，且**不得新增第二个极性开关**（两处反转互相抵消，是 bug 温床）。
- **需要答复**：Q4 的对调是否为最终版？PA6 的晶振冲突是否已实测确认？

## 3. 施工约束（待解除阻断后生效，先行记录）

- **原子性**：编码器与灰度**必须同一次施工、同一提交**。分批改必然出现 PB7/PB9/PB10/PB11 被两个模块同时声明的 syscfg 冲突，中间态无法编译/烧录。
- **事实来源唯一**：一切以 `board.syscfg` 为准，改配置只能经 SysConfig 产出；不得手改 `ti_msp_dl_config.*` 生成物。
- **公共 API 不变**：`Encoder_Init/Update/GetSnapshot`（P2 已验收）与 `BoardGpio_GetEncoderRawSnapshot` 的签名与语义**不得因引脚迁移而改动**——迁移是配置层与 `board_gpio` 绑定层的事，不得上溯到 Driver 公共面。灰度同理（`TRACK_SENSOR_COUNT=12` 不变）。
- **左右轮归属必须证明**：需要一条独立证据行证明 `QEI_LEFT` 绑定 TIMG9、`QEI_RIGHT` 绑定 TIMG8，且 `s_direction_sign` 已按新接线重新标定。**主机测试无法证明引脚接线**——此项只能由用户板上实测（左轮正转 → 左轮计数为正）闭环，Codex 只验证配置与代码的一致性。
- **禁止**：借引脚迁移之名改 Encoder/灰度的算法、滤波或标定参数；新增第二个极性反转开关。
