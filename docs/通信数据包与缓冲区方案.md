# 通信数据包与缓冲区方案（WL2，2026-07-23 定案）

> 一句话：**丢包不是靠一个大缓冲区消灭的，是靠「每一层丢失都有计数器 + 事件类端到端
> 重传」消灭的。** 本文是全仓串口/无线通信的唯一方案文档：三层模型、通用缓冲区范式、
> 数据包协议 v2、以及 systick/中断表/时序/主频/协议五因素逐项过账。

## 1. 三层模型——每层丢失都可查询

| 层 | 所有者 | 会丢什么 | 计数器（查询入口） |
|---|---|---|---|
| 端口字节层 | `driver/board_uart/*` | RX FIFO 满丢字节；TX 环满拒帧 | `*_GetRx/TxOverflowCount`（UartDiag 条目一页全查） |
| 编解码帧层 | `driver/uart_wireless` 等 | CRC 坏帧、越界帧、未知 type、seq 空洞 | `crc_error/unknown_type/ur_gap/ur_dup`（LinkTest） |
| 投递语义层 | `app/service/link` | 事件重试耗尽、状态被覆盖（设计内） | `retx/delivered/ev_fail`（LinkTest） |

诊断路径固定三步：**先看 UartDiag（字节层丢没丢）→ 再看 LinkTest ur_gap（空中丢了几帧）
→ 最后看 ev_fail（有没有指令没送到）**。任何"感觉丢包"10 秒内定位到层。

VOFA 调参链路专项：丢帧的两个 MCU 侧母因（RX 命令丢字节 / TX 遥测环满拒帧）都在
UartDiag 的 ch0/ch1；PC 侧显示丢帧不在此列（VOFA 上位机自身掉点，用 100ms 方波标定
时间轴甄别——测量纪律 §4）。帧层可视化技术：任何条目可自愿注册一个自增计数通道，
波形锯齿断点=该帧丢失（登记不强制）。

## 2. 通用缓冲区范式（T-VTXQ1 pattern，已被 28 用例证明）

**结论：范式复制，不做共享模块。** 四个现役端口（vofa/vision/stepmotor/imu）已测绿
不重构（手术纪律）；新端口（wireless 真端口）照抄 `vofa_uart.c`，规格已写进
`wireless_uart.h` 头注。

范式六要素：

1. **RX 私有字节环**：ISR 只做 `IsrPushByte`（搬运+满则 overflow_count++），
   解析一律任务态（V09 边界）；
2. **TX 软件字节环**：`TryWrite` 忙时**入队**而非丢帧；只有环满才拒绝并计数——
   拒绝对上层可见（返回 false），永不静默丢；
3. **DMA kick 链**：`TryWrite` 持锁 kick / `IsrTxDone` 清 busy 后续段；busy 标志是
   DMA 忙的唯一权威；环容量 ≤ DMA 发送缓冲（编译期断言防越界）；
4. **短临界区**：PRIMASK 开关中断只包住指针操作（微秒级），不包 CRC/组帧；
5. **静态所有权**：FIFO 全部 `static` 于端口 .c 内，无跨层暴露、无回调注册
   （中断分派固定写在 mspm0_runtime，V02 关闭先例）；
6. **容量公式**：`bytes = (baud/10) × 2×服务周期ms × 安全系数2`，取整到 2 的幂。

| 端口 | 波特率 | 服务周期 | 公式值 | 现值/规格 |
|---|---|---|---|---|
| VOFA (UART5) | 230400 | 5ms | 461 | RX 512 / TX 512（已落地） |
| Stepmotor (UART7) | 256000 | 请求-应答锁步 | — | RX FIFO；TX 单帧在途（协议锁步，队列反而破坏节奏，**有意不加**） |
| Vision | 230400 | 10ms | 922* | RX FIFO；TX 低频握手轮询式（*坐标帧 11B/帧实际远低于线速上限） |
| IMU (UART3) | 230400 | 10ms | 922* | RX FIFO 纯 IRQ（*同上，实际帧流 ~66B/10ms） |
| Wireless（待定案） | 115200 | 10ms | 461 | RX 512 / TX 512（规格已冻结在 wireless_uart.h） |

## 3. 数据包协议 v2（帧格式见 uart_wireless.h，此处只记设计依据）

`0xA5 0x5A | len | type | seq | payload(≤32) | CRC16-MODBUS LE`（CRC 含 seq）

用户五要求 → 机制映射：

- **校验**：CRC16-MODBUS（覆盖 len+type+seq+payload，坏一位整帧拒收）；
- **筛选**：双头猎取自同步 + len≤32 白名单 + type 白名单（未知 type 即使 CRC 合法也
  拒收并单独计数——协议失配探针，不污染活性）；
- **排除**：seq 去重——不可靠流同 seq 丢弃计数；事件流同 seq **重发 ACK 但不重复投递**
  （对端 ACK 丢失场景，幂等）；
- **挑选**：状态帧信箱「后到覆盖先到」（过期状态无价值）；事件帧深 4 FIFO 队列全收；
  **队列满不 ACK**——宁可让对端重传，绝不假确认后丢弃；
- **梯队**：TX 优先序 ACK（Poll 内即回）＞事件重发＞状态＞心跳；任一成功数据 TX
  抑制本拍心跳（对端活性窗口已被刷新，重复心跳是纯浪费）。

**「每帧不丢」的工程分解**（为什么不是"全部重传"）：

- 事件类（指令：开始/停止/到位/装载完成）→ stop-and-wait 必达：单在途 + ACK +
  40ms×8 重传 + 收端去重。**要么送达（delivered++），要么 320ms 内明确失败
  （ev_fail++）**——没有第三种"悄悄丢了"。单在途=天然保序，指令类无需滑动窗口
  （帧短、低频，窗口机制的复杂度买不来任何吞吐收益——简单性优先）。
- 状态类（位姿/进度周期播报）→ 重传旧状态毫无意义（新帧马上来且更新），改为
  **丢失可测**：共享 seq 的 gap 账 = 空中丢包数直读，心跳空闲期也在持续探测丢包率。
- ESP-NOW 跳的丢包对 UART 字节层不可见，**这就是 ACK 必须端到端（MCU↔MCU）而不能
  依赖任何一段字节缓冲的根本原因**。

## 4. 五因素过账（A~E）

**A. systick / 调度节拍**：时间轴唯一源=调度器注入的 `now_ms`（systick 1ms 累加），
Driver 层零时间轴（Q2/Q7 分层）。所有通信节拍都是 10ms 泵的整数倍：link 门控 10ms、
事件重试 4 tick=40ms、心跳 20 tick=200ms、活性 60 tick=600ms。缓冲容量公式里的
"服务周期"就是这个 10ms——**改泵周期必须同步复核 §2 表**（公式在，重算 30 秒）。

**B. 中断表**：现役 UART 中断=DMA_IRQHandler（stepmotor/vofa/vision 三对 RX/TX 通道）
+ 各 UART 实例 IRQ（含 IMU 纯 IRQ）+ GROUP1（仅按键）。全部固定分派、无回调表（V02）、
无嵌套优先级需求——每个 ISR 只做读清+搬字节+计数（微秒级），不存在互相饿死。
wireless 真端口接入=同表加两行固定分派（RX 字节→IsrPushByte、TX done→IsrTxDone），
不新增任何优先级设计。

**C. 时序账**：115200 下最长帧 39B≈3.4ms 线时间；一拍梯队全发（ACK7+事件39+状态39+
心跳7=92B≈8ms）< 10ms 泵周期——**不存在持续积压**，512B TX 环吸收的是瞬时突发。
事件最坏判定时间=首发+8×40ms=320ms < 600ms 活性窗口（先知道"事件失败"，后知道
"链路死了"，语义正确）。VOFA 遥测：10ms 一帧 ~100B@230400≈4.3ms < 周期，环内稳态
占用为零。ESP-NOW 单跳延迟 ~2-4ms，对 40ms 重试节拍透明。

**D. 主频账（80MHz 上限）**：全部五路串口同时满线速 ≈106KB/s ≈ 10.6 万次/秒字节
ISR × ~60 cycle ≈ 6.4M cycle/s ≈ **8% CPU（理论天花板，实际帧流 <1%）**。CRC16
逐位法 40B 帧 ≈3200 cycle ≈40µs（任务态）——一拍梯队全发也只 ~160µs。**结论：主频
不是通信瓶颈，禁止以"性能"为由上查表 CRC/DMA 链表等复杂度**（性能改动需基线，§12）。

**E. 协议**：v2 对 §35.1 的修订=+1 seq 字节、+2 type；对端（ESP32-C3 透传固件、
对面车 MCU）尚不存在，零兼容负担。ESP-NOW 250B 上限 >39B 单帧，透传固件按字节流
转发即可，自同步分帧天然容忍空中任意重新分块。VOFA JustFloat 协议不动（PC 工具
约束），其丢帧治理=计数器外显（§1）而非改协议。

## 5. 现场排障速查（人话版）

| 症状 | 先看 | 结论 |
|---|---|---|
| VOFA 波形有断点 | UartDiag ch0/ch1 | 涨=MCU 丢字节（查线/降载）；不涨=PC 侧掉点（标时间轴） |
| 调参命令没反应 | UartDiag ch0 | 涨=RX 丢字节；不涨=命令格式/通道映射错 |
| 无线"感觉丢包" | LinkTest ch3 | 涨=空中丢（挪 ESP32 位置/降速率）；不涨=没丢，是消费侧没读 |
| 指令没执行 | LinkTest ch6 | 涨=重传 8 次全失败（链路真断）；ch5 涨=送到了，是对端逻辑问题 |
| 双机 ch2 猛涨 | — | ESP32 透传波特率不匹配 |

一句话总结：**状态类比新鲜、事件类必达、每层丢失都有表可查——从此"丢帧"不再是悬案，
是一个 10 秒能定位到层的读数。**
